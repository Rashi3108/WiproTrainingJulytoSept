What is Code Quality?

üìã Definition

Code quality refers to how well-written, maintainable, and reliable source code is. It encompasses multiple dimensions that affect the software's long-term success.

üéØ Key Dimensions of Code Quality

1. üêõ Reliability

  • Bugs: Code that is likely to behave unexpectedly
  • Error Handling: Proper exception management
  • Edge Cases: Handling of boundary conditions

2. üõ°Ô∏è Security

  • Vulnerabilities: Code that can be exploited by attackers
  • Security Hotspots: Code that requires security review
  • Data Protection: Secure handling of sensitive information

3. üîß Maintainability

  • Code Smells: Code that is hard to understand or modify
  • Complexity: Cyclomatic and cognitive complexity
  • Duplications: Repeated code blocks

4. üß™ Testability

  • Test Coverage: Percentage of code covered by tests
  • Unit Tests: Individual component testing
  • Integration Tests: System interaction testing

‚ö†Ô∏è Consequences:
  • Increased development time and costs
  • Higher bug rates and production issues
  • Security vulnerabilities and data breaches
  • Difficulty in adding new features
  • Developer frustration and turnover

Key SonarQube Concepts

üêõ 1. Bugs

Code that is wrong or will behave unexpectedly

Characteristics:

  • Likely to cause runtime errors or unexpected behavior
  • Should be fixed immediately
  • Examples: Null pointer dereferences, resource leaks, infinite loops

Example - Null Pointer Bug:

// Bug: Potential NullPointerException
public String getUserName(User user) {
    return user.getName().toUpperCase(); // user could be null
}

// Fixed version:
public String getUserName(User user) {
    if (user != null && user.getName() != null) {
        return user.getName().toUpperCase();
    }
    return "Unknown";
}

üëÉ 2. Code Smells

Code that is confusing, hard to maintain, or violates coding standards

Types of Code Smells:

  • Maintainability: Long methods, large classes, complex conditions
  • Reliability: Empty catch blocks, unused variables
  • Readability: Poor naming, missing documentation

Example - Long Method Smell:

// Code Smell: Method too long (>20 lines)
public void processOrder(Order order) {
    // 50+ lines of code doing multiple things
    validateOrder(order);
    calculateTax(order);
    applyDiscounts(order);
    updateInventory(order);
    sendNotification(order);
    logTransaction(order);
    // ... more code
}

// Better: Extract methods
public void processOrder(Order order) {
    validateOrder(order);
    calculateOrderTotal(order);
    fulfillOrder(order);
    notifyCustomer(order);
}

üõ°Ô∏è 3. Security Vulnerabilities

Code that can be exploited by hackers

Common Vulnerability Types:

  • Injection: SQL injection, XSS, command injection
  • Authentication: Weak passwords, session management
  • Cryptography: Weak encryption, hardcoded secrets
  • Input Validation: Unvalidated user input

Example - SQL Injection Vulnerability:

// Vulnerability: SQL Injection
String query = "SELECT * FROM users WHERE id = " + userId;
Statement stmt = connection.createStatement();
ResultSet rs = stmt.executeQuery(query);

// Secure version: Prepared Statement
String query = "SELECT * FROM users WHERE id = ?";
PreparedStatement pstmt = connection.prepareStatement(query);
pstmt.setInt(1, userId);
ResultSet rs = pstmt.executeQuery();

üìã 4. Code Duplications

Identical or similar code blocks repeated across the codebase

Types of Duplication:

  • Exact Duplication: Identical code blocks
  • Structural Duplication: Same structure, different variables
  • Functional Duplication: Different code achieving same result

Impact:

  • Increases maintenance effort
  • Higher risk of inconsistent bug fixes
  • Larger codebase and memory footprint

üí∞ 5. Technical Debt

The cost of additional rework caused by choosing quick solutions over better approaches

SonarQube Technical Debt Calculation:

üìä Debt Ratio

Ratio between cost to fix issues and cost to develop the application

‚è±Ô∏è Remediation Time

Estimated time needed to fix all maintainability issues

üéØ SQALE Rating

A-E rating based on technical debt ratio

üìà Trend Analysis

Track debt evolution over time

Technical Debt Ratings:

  • A: ‚â§ 5% - Excellent
  • B: 6-10% - Good
  • C: 11-20% - Moderate
  • D: 21-50% - Poor
  • E: > 50% - Very Poor

SonarQube Dependencies & Requirements

üîß Core Dependencies

SonarQube requires several key components to function properly. Understanding these dependencies is crucial for successful installation and operation.

‚òï 1. Java Runtime Environment (JRE/JDK)

üìã Requirements

Version: Java 11 or Java 17
Type: OpenJDK or Oracle JDK
Memory: Minimum 2GB heap

üéØ Why Java?

SonarQube is built entirely in Java. The server, web interface, and analysis engine all run on the JVM.

Java Usage in SonarQube:

  • Web Server: Runs the SonarQube web application
  • Compute Engine: Processes analysis reports
  • Background Tasks: Handles project indexing and cleanup
  • Plugin System: Loads and executes language analyzers

üîç 2. Elasticsearch (Embedded)

üìã Details

Version: Embedded in SonarQube
Purpose: Search and indexing engine
Memory: 512MB minimum

üéØ Why Elasticsearch?

Provides fast search capabilities for issues, rules, and project data across large codebases.

Elasticsearch Usage:

  • Issue Search: Fast searching through millions of code issues
  • Rule Indexing: Quick access to quality rules and profiles
  • Project Navigation: Efficient browsing of large projects
  • Dashboard Queries: Real-time data aggregation for metrics

üóÑÔ∏è 3. Database (Optional but Recommended)

üìã Supported Databases

Production: PostgreSQL, Oracle, SQL Server
Development: H2 (embedded)
Recommended: PostgreSQL

üéØ Why Database?

Stores all persistent data including issues, metrics, user settings, and historical analysis data.

Database Usage:

  • Issue Storage: All detected bugs, vulnerabilities, and code smells
  • Metrics History: Historical trends and quality gate results
  • User Management: Authentication, permissions, and preferences
  • Configuration: Quality profiles, rules, and project settings

üîí Security Dependencies

Operating System Requirements:

  • File Permissions: Dedicated user account (sonarqube)
  • System Limits: Increased file descriptors and processes
  • Kernel Parameters: Memory mapping limits for Elasticsearch
  • Firewall: Proper port configuration

Critical System Settings:

# File descriptor limits
echo "sonarqube soft nofile 65536" >> /etc/security/limits.conf
echo "sonarqube hard nofile 65536" >> /etc/security/limits.conf

# Process limits
echo "sonarqube soft nproc 4096" >> /etc/security/limits.conf
echo "sonarqube hard nproc 4096" >> /etc/security/limits.conf

# Kernel parameters for Elasticsearch
echo "vm.max_map_count=262144" >> /etc/sysctl.conf
echo "fs.file-max=65536" >> /etc/sysctl.conf
‚ö†Ô∏è Important: These system-level configurations are required because:
  • Elasticsearch needs high file descriptor limits for indexing
  • Java applications require sufficient process limits
  • Memory mapping is essential for search performance

SonarQube Metrics Explained

üêõ 1. Bugs

What SonarQube Considers as Bugs:

Severity Description Example Impact
CRITICAL Will likely cause runtime errors Null pointer dereference Application crash
MAJOR May cause unexpected behavior Resource not closed Memory leaks
MINOR Could lead to issues Unused variables Code confusion

Spring Boot Project - Typical Bug Examples:

// Potential bug: Resource leak
@GetMapping("/file")
public String readFile() {
    FileInputStream fis = new FileInputStream("file.txt");
    // Bug: Stream not closed - SonarQube will flag this
    return fis.toString();
}

// Fixed version:
@GetMapping("/file")
public String readFile() {
    try (FileInputStream fis = new FileInputStream("file.txt")) {
        return fis.toString();
    } catch (IOException e) {
        return "Error reading file";
    }
}

üëÉ 2. Code Smells

Maintainability Issues in Spring Boot:

üîß Method Complexity

Methods with high cyclomatic complexity (>10)

üìè Method Length

Methods longer than 20 lines

üèóÔ∏è Class Size

Classes with too many methods or lines

üìù Documentation

Missing JavaDoc for public methods

Example Code Smell:

// Code Smell: Method too complex
@GetMapping("/process")
public ResponseEntity<String> processRequest(@RequestParam String type) {

        if (condition1) {
            if (condition2) {
                // Deep nesting - SonarQube will flag this
                return ResponseEntity.ok("Result A");
            }
        }
    } else if (type.equals("B")) {
        // More complex logic...
    }
    return ResponseEntity.badRequest().body("Invalid type");
}

üõ°Ô∏è 3. Security Vulnerabilities

Common Spring Boot Security Issues:

  • SQL Injection: Unsafe database queries
  • XSS: Unescaped user input in responses
  • Path Traversal: Unsafe file operations
  • Weak Cryptography: Insecure random number generation

Example Security Vulnerability:

// Security Vulnerability: SQL Injection
@GetMapping("/user/{id}")
public User getUser(@PathVariable String id) {
    String sql = "SELECT * FROM users WHERE id = " + id;
    // Vulnerable to SQL injection - SonarQube will flag this
    return jdbcTemplate.queryForObject(sql, User.class);
}

// Secure version:
@GetMapping("/user/{id}")
public User getUser(@PathVariable String id) {
    String sql = "SELECT * FROM users WHERE id = ?";
    return jdbcTemplate.queryForObject(sql, User.class, id);
}
